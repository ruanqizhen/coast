## 1. 产品概述

### 1.1 定位

**Coast** 是一款运行于桌面浏览器的 2.5D / 真 3D 主题公园模拟经营游戏。玩家从一块 64×64 网格的空地起步，通过规划道路、摆放设施、雇佣员工和优化定价，将公园逐步经营至五星评级。

**核心差异化主张（USP）：**

| 差异点 | 说明 |
| --- | --- |
| Web 即开即玩 | 无需安装，URL 分享，5 秒内进入游戏 |
| 真实游客 AI | FSM + 效用评分，每位游客有独立需求与决策 |
| 可自定义过山车 | 分段拼装轨道，物理校验 G 力，AI 安全检测 |
| 轻量化运营 | 15–30 分钟碎片化游玩，随时存档 |

### 1.2 竞品参照

| 参照作品 | 借鉴方向 | 差异化方向 |
| --- | --- | --- |
| 《过山车大亨 2》 | 设施深度、经济系统 | Web 平台、现代 UI |
| 《主题公园》 | 游客 AI 设计 | 3D 渲染、分享机制 |
| 《双点医院》 | 美术风格、轻松基调 | 主题公园垂直品类 |
| 《Parkitect》 | 过山车编辑器 | 浏览器可访问性 |

### 1.3 平台与分辨率

- **主要平台**：Chrome / Edge / Safari 桌面端
- **最低分辨率**：1280 × 720
- **移动端**：Phase 3 后提供只读查看模式（不支持建造操作）
- **推荐配置**：4 核 CPU + 8 GB RAM，集成显卡可运行

## 2. 目标用户

### 2.1 用户画像

**用户群 A · 怀旧玩家**
- 年龄：30–45 岁，有《过山车大亨》或《主题公园》经历
- 诉求：还原经典深度玩法，不希望学习曲线过陡
- 痛点：经典游戏已停更，现代作品需要下载安装

**用户群 B · 休闲经营玩家**
- 年龄：20–35 岁，喜欢《星露谷》《双点》类型
- 诉求：碎片化时间可玩，有清晰的正反馈
- 痛点：同类游戏单局时间过长，难以中途退出

**用户群 C · 创作分享者**
- 年龄：18–30 岁，活跃于 Reddit / B 站 / Twitter
- 诉求：可截图/录屏展示公园，获得社区认可
- 痛点：截图质量差，难以展示整体布局

### 2.2 核心用户场景

```
场景 A：下班后 30 分钟快速游玩
  → 上午保存的公园自动恢复，直接继续经营

场景 B：周末 2 小时深度规划
  → 拆除旧区域重新布局，设计一条新过山车线路

场景 C：完成里程碑后分享截图
  → 一键生成鸟瞰截图并复制分享链接
```

## 4. 核心游戏循环

### 4.1 宏观循环

```
建造 → 吸引游客 → 产生收入与满意度 → 研发解锁 → 扩张地块 → 建造...
```

每个大循环对应若干游戏日，玩家通过资源积累推动公园成长。

### 4.2 时间系统

| 参数 | 数值 |
| --- | --- |
| 1 游戏日 | 现实 60 秒 |
| 速度选项 | 暂停 / ×1 / ×2 / ×4 |
| 月结算 | 每 30 游戏日触发一次收支结算 |
| 年结算 | 每 12 个游戏月触发一次星级评定 |

### 4.3 微观循环（单日视角）

```
游客入园
  │
  ├── 需求评估（每 3 秒）
  │     ├── 查找设施 → 排队 → 游玩 → 满意度 ↑，收入 ↑
  │     ├── 查找餐饮 → 消费 → 饥渴槽 ↑，收入 ↑
  │     └── 查找休息区 → 恢复疲劳/恶心
  │
  ├── 满意度累积 → 影响口碑 → 次日游客量
  │
  └── 设施磨损 → 修理工响应 → 维护费支出
```

### 4.4 星级评定规则

| 星级 | 满意度要求 | 游客数要求 | 解锁内容 |
| --- | --- | --- | --- |
| ★☆☆☆☆ | ≥ 30 | ≥ 50 | 基础设施全集 |
| ★★☆☆☆ | ≥ 50 | ≥ 150 | 研发树第二排 |
| ★★★☆☆ | ≥ 65 | ≥ 300 | 地块扩张至 96×96 |
| ★★★★☆ | ≥ 78 | ≥ 500 | 弹射过山车、暗黑骑乘 |
| ★★★★★ | ≥ 90 | ≥ 800 | 地块扩张至 128×128，成就徽章 |

## 5. 系统详解

### 5.1 建设系统

#### 5.1.1 地块与网格

- **网格单位**：1 格 = 2 米
- **起始地块**：64×64 格（约 128m × 128m）
- **最大地块**：128×128 格（阶段解锁）
- **扩张费用**：每次扩张 16 格边长，费用 $2000，需对应星级解锁
- **放置方向**：四方向，支持 90° 旋转

#### 5.1.2 放置规则

| 规则 | 说明 |
| --- | --- |
| 道路连通 | 所有设施必须与道路网络相连，否则无法放置 |
| 地形限制 | 大型设施（占地 > 3×3）需地形平整度 ≥ 80% |
| 员工通道 | 员工专用通道不纳入游客寻路网格，对游客不可见 |
| 重叠检测 | 实时判定，放置时半透明预览，红色 = 不可放，绿色 = 可放 |
| 拆除退款 | 拆除已建设施退还建造价 40%，首月内退还 70% |

#### 5.1.4 道路系统

| 类型 | 宽度 | 造价 | 说明 |
| --- | --- | --- | --- |
| 普通道路 | 1 格 | $5/格 | 游客通行，纳入寻路网格 |
| 宽幅道路 | 2 格 | $9/格 | 拥挤度阈值 ×2，适合主干道 |
| 员工通道 | 1 格 | $3/格 | 员工专用，游客不可见、不可走 |

寻路更新：铺路或拆路后 0.2 秒内触发增量 A* 权重重算，只更新受影响节点。

#### 5.1.5 过山车轨道编辑器（Phase 3）

轨道段类型：

| 段类型 | 刺激值加成 | 晕眩值加成 | 故障率加成 |
| --- | --- | --- | --- |
| 直线段 | 0 | 0 | 0 |
| 爬升段 | +0.5/段 | +0.2/段 | +0.001 |
| 俯冲段 | +1.2/段 | +0.8/段 | +0.003 |
| 回环段 | +2.0/个 | +1.5/个 | +0.008 |
| 超级回环（研发） | +3.0/个 | +2.0/个 | +0.015 |

安全校验规则：
- 轨道必须形成闭合回路，否则无法运营
- AI 自动计算各段 G 力，任意段超过 **6G** 禁止运营
- 玩家可手动调整段坡度（步进 5°），超限时实时红色警告
- 轨道总长度影响单次乘坐时长（最短 30 秒，最长 5 分钟）

### 5.2 游客 AI 系统

#### 5.2.1 设计原则

采用 **FSM（有限状态机）+ 效用评分**架构，而非行为树，以降低单帧计算量。每位游客是独立的 Entity，决策在 Web Worker 中异步执行，主线程只接收位置与状态更新。

#### 5.2.2 个体属性（入园时随机生成）

| 属性 | 范围 | 说明 |
| --- | --- | --- |
| 资金 | $20–$80 | 决定消费上限 |
| 年龄组 | 儿童 / 青少年 / 成人 / 家庭 | 影响设施偏好权重 |
| 刺激偏好 | 1–10 | 越高越倾向于高刺激设施 |
| 晕眩耐受 | 1–10 | 低于设施晕眩值时恶心累加速度加快 |
| 排队耐心 | 60–120 秒 | 超时离开并扣满意度 |
| 消费意愿 | 0.5–1.5 | 乘数，影响对价格的敏感度 |

年龄组偏好修正：

| 年龄组 | 刺激设施权重 | 温和设施权重 | 商业消费倾向 |
| --- | --- | --- | --- |
| 儿童 | 0.4 | 1.8 | 1.2（零食优先） |
| 青少年 | 1.5 | 0.8 | 1.0 |
| 成人 | 1.0 | 1.0 | 1.3（餐饮优先） |
| 家庭 | 1.1 | 1.3 | 1.5 |

#### 5.2.3 需求槽系统

需求槽每秒自然衰减（数值越高越急迫），触发对应寻路行为：

| 需求 | 范围 | 衰减/秒 | 行为触发阈值 | 触发行为 |
| --- | --- | --- | --- | --- |
| 饥饿 | 0–100 | -0.8 | < 30 | 寻找餐饮摊位 |
| 口渴 | 0–100 | -1.2 | < 25 | 寻找饮料摊位 |
| 如厕 | 0–100 | -0.5 | > 70 | 寻找洗手间 |
| 疲劳 | 0–100 | -0.3 | > 60 | 寻找长椅休息 |
| 恶心 | 0–100 | — | > 50 | 寻找长椅或就地呕吐；>80 前往急救站 |
| 乐趣 | 0–100 | -0.6 | < 40 | 主动寻找设施 |

恶心累加触发器：
- 乘坐设施后：`恶心 += 设施晕眩值 × (1 - 游客晕眩耐受/10)`
- 连续乘坐 2 个高晕眩设施（>5）：`恶心 += 20`（额外惩罚）

#### 5.2.4 决策逻辑

```
每 3 秒触发一次决策评估：

1. 扫描 5 格内可达目标（设施、餐饮、休息区）
2. 对每个目标计算效用分：
   效用 = 吸引力 × 偏好匹配系数 / (距离 × 排队时长权重) × 需求紧迫系数
3. 选择效用最高的目标
4. 调用 A* 寻路，生成路径
5. 排队时每 10 秒检查剩余耐心：
   - 耐心耗尽 → 放弃目标，满意度 -5，重新决策
   - 进入设施 → 游玩，满意度 += 效果值
```

```
[入园] → [空闲/决策] ←──────────────────┐
              │                            │
      ┌───────┴────────┐                  │
      ▼                ▼                  │
  [寻路中]         [需求紧急]             │
```

#### 5.2.6 游客上限与生成控制

| 参数 | 数值 |
| --- | --- |

### 5.3 经济与财务系统

#### 5.3.1 收入来源

**门票（可选模式）**

| 模式 | 门票价 | 说明 |
| --- | --- | --- |
| 免费入园 | $0 | 游客量上限更高，靠设施和餐饮盈利 |
| 付费入园 | $10–$50（玩家设定） | 每涨 $10 游客量减少约 8% |

**单次乘坐票价**

默认范围 $3–$8，玩家可单独调整每个设施定价。价格弹性公式：

```
游客接受率 = 1 - 0.4 × (设定价 - 默认价) / 默认价
```

若设定价超出游客资金上限，该游客跳过该设施。

**商品销售**

```
销量 = 基础销量 × [1 - 0.5 × (设定价 - 默认价) / 默认价]
利润 = 销量 × (设定价 - 成本)
成本 = 设定价 × 30%
```

#### 5.3.2 支出明细

| 支出类型 | 计算方式 | 备注 |
| --- | --- | --- |
| 建设费 | 一次性，放置时扣除 | 拆除退 40%（首月 70%） |
| 月度维护 | 建造价 × 2% / 月 | 故障率高的设施维护费更高 |
| 员工工资 | 见员工系统 | 每月结算 |
| 贷款利息 | 本金 × 1.5% / 月 | 资金不足时可申请贷款（上限 $5,000）|
| 研发投入 | 玩家每月自行设定 $0–$500 | — |

#### 5.3.3 满意度计算

```
满意度（0–100）=
  0.4 × 游玩体验分   （设施平均评分 × 乘坐率）
+ 0.3 × 环境清洁分   （垃圾密度、清洁工覆盖率）
+ 0.2 × 性价比分     （票价 / 游客钱包比）
+ 0.1 × 员工服务分   （演艺覆盖率、急救响应速度）
```

| 满意度区间 | 影响 |
| --- | --- |
| ≥ 80 | 口碑传播，次日游客量 +20% |
| 60–79 | 正常运营 |
| 40–59 | 游客开始主动离园（离园率 +10%/小时） |
| < 40 | 游客大量离园，不再主动入园 |

#### 5.3.4 数据面板

- **折线图**：近 12 个游戏月的日收入、游客数、满意度趋势
- **柱状图**：各设施收入占比、各类别支出占比
- **实时仪表**：当日净利润、游客峰值、设施利用率

### 5.4 员工系统

#### 5.4.1 设计原则

员工不可直接操控路径，玩家只能划定 8×8 格的**巡逻区域**。员工 AI 在 Web Worker 中运行任务队列，主线程只接收位置更新。

#### 5.4.2 员工类型

| 员工 | 月薪 | 主要职责 | 工作速度 |
| --- | --- | --- | --- |
| 清洁工 | $60 | 清扫垃圾和呕吐物，清倒垃圾桶 | 1.2 格/秒 |
| 修理工 | $90 | 巡检设施，修复故障 | 1.0 格/秒 |
| 保安 | $70 | 阻止破坏行为，维护秩序 | 1.4 格/秒 |
| 演艺人员 | $50 | 提升周边游客心情 | 0.8 格/秒 |

#### 5.4.3 员工行为规则

**清洁工**（优先级从高到低）
1. 呕吐物（满意度惩罚最高）
2. 垃圾桶已满（触发率 = 容量 >90%）
3. 普通垃圾
4. 花坛浇水（环境分加成）

**修理工**
- 每 60 秒对巡逻区内设施进行一次状态巡检
- 故障率计算：`故障率 = 设施年龄 × 0.001 + 累计使用次数 × 0.0005 + 轨道复杂度加成`
- 设施故障后立即停运，修理耗时 15 秒，修理完成后自动恢复
- 可升级工具箱（研发树）将修理时间缩短至 8 秒

**保安**
- 破坏行为触发率：`破坏率 = 区域拥挤度 × 0.01 × 青少年占比`
- 响应时间 < 30 秒内阻止破坏，否则设施耐久 -10

**演艺人员**
- 半径 3 格内所有游客：乐趣 +0.5/秒，疲劳回复 +0.2/秒
- 每 20 分钟需要休息 5 分钟（体力槽设计）

## 8. 技术架构

### 8.1 技术选型

| 层 | 技术 | 版本 | 选型理由 |
| --- | --- | --- | --- |
| 渲染引擎 | Babylon.js | 7.x | 内置 GUI、物理引擎、Instanced Mesh、WebGPU 支持，适合复杂 Tycoon 场景 |
| UI 框架 | React | 18 | 游戏外层 UI（HUD、面板、建造栏） |
| 状态管理 | Zustand | 4.x | 轻量，避免 Redux 样板代码 |
| 并发计算 | Web Worker | 原生 | AI 决策、寻路、员工逻辑 |
| 数据存储 | IndexedDB | 原生 | 存储完整存档 JSON |
| 构建工具 | Vite | 5.x | 快速 HMR，按需分包 |

### 8.2 架构分层

```
┌─────────────────────────────────────────┐
│  Presentation Layer                     │
│  React HUD + Babylon.js Canvas          │
├─────────────────────────────────────────┤
│  Game State Layer                       │
│  Zustand Store（parks, settings, ui）   │
├─────────────────────────────────────────┤
│  Simulation Layer（Web Worker）         │
│  AISystem / EconomySystem               │
│  PathfindingSystem / StaffSystem        │
├─────────────────────────────────────────┤
│  Data Layer                             │
│  IndexedDB（coast_save）                │
│  localStorage（最后存档 ID）            │
└─────────────────────────────────────────┘
```

### 8.3 ECS 模式

| 层级 | 内容 |
| --- | --- |
| Entity | 游客、员工、设施 |
| Component | Position, Needs, Wallet, RideStats, BreakdownTimer |
| System | AISystem, EconomySystem, PathfindingSystem, StaffSystem |

System 均运行于 Web Worker，每帧结果通过 `postMessage` 传回主线程，主线程只负责渲染与输入。

### 8.4 寻路设计

- 算法：**增量 A\***（只重算受影响节点）
- 分片执行：每帧最多计算 **20 个游客路径**，其余加入队列
- 网格节点权重：道路 1，拥挤区域 1.5–3，障碍物无穷大
- 更新触发：铺路/拆路后 0.2 秒内触发局部重算

### 8.5 性能优化策略

| 策略 | 说明 |
| --- | --- |
| Instanced Mesh | 游客 3 种模型，员工 1 种，各自 Instance，1000 人只需 4 次 draw |
| 对象池 | 垃圾、呕吐物预生成 200 个，复用避免 GC |
| LOD | 镜头高度 > 30 格时，切换简模并关闭骨骼动画 |
| 视锥剔除 | 摄像机视野外的对象不提交渲染 |
| 分片寻路 | 每帧上限 20 个路径计算，其余排队 |
| 按需更新满意度 | 满意度每 5 秒重算一次，非逐帧 |
| 批量 DOM 更新 | HUD 数据每 500ms 同步一次，非逐帧 |

### 8.6 数据存储

- 数据库名：`coast_save`，表：`parks`，字段：`id`, `name`, `data`（JSON），`updatedAt`
- `localStorage` 存储最近打开的存档 ID，用于快速续玩
- 自动存档：每 5 分钟触发，存档前先显示 0.5 秒"保存中..."提示
- 存档大小估算：500 游客场景下约 200KB，压缩后 < 50KB

## 11. 风险与应对

| 风险 | 可能性 | 影响 | 应对策略 |
| --- | --- | --- | --- |
| 1,000 游客性能不达标 | 高 | 高 | 优先 Instanced Mesh + Web Worker；若仍不足，降低硬上限至 800 |
| 过山车编辑器复杂度超期 | 中 | 中 | Phase 2 结束前确认基础编辑器原型，Phase 3 只做打磨 |
| IndexedDB 在 Safari 存在限制 | 中 | 低 | 提供 JSON 文件导出/导入作为备选存档方案 |
| A* 寻路在大地图性能下降 | 中 | 中 | 分层寻路（宏观网格 + 局部网格）作为备选方案 |
| 玩家上手门槛过高 | 中 | 高 | 强制新手引导（前 5 分钟），可跳过但默认开启 |
| WebGPU 兼容性不足 | 低 | 低 | 默认 WebGL 2.0，WebGPU 作为可选高画质模式 |
